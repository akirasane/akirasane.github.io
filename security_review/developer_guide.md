# คู่มือการใช้งานสำหรับนักพัฒนา (Developer Integration & User Guide)
## ระบบ Two-Layer Security Review (SAST & LLM Orchestration)

---

## 1. การทำงานของระบบเบื้องหลัง (How It Works)

เมื่อคุณเปิด Merge Request (MR) หรือทำการอัปเดตโค้ดเข้าสู่ GitLab ระบบความปลอดภัยสองชั้นจะทำงานโดยอัตโนมัติดังนี้:
1. **ด่าน 1 (SAST):** SonarQube และ Snyk จะตรวจสอบโค้ดของคุณเพื่อหาความบกพร่องเชิงโครงสร้างและช่องโหว่พื้นฐาน จากนั้นส่งรายงานเป็นไฟล์ SARIF
2. **ด่าน 2 (AI Review):** AI Bot (ผ่าน MCP Server) จะอ่านรายงานความผิดพลาดร่วมกับส่วนโค้ดที่ต่างกัน (Diff) เพื่อทำการวิเคราะห์บริบทเชิงตรรกะ
3. **การส่งรายงาน:** Bot จะทำการเขียนข้อความวิเคราะห์ความเสี่ยงและแนวทางแก้ไขตรงบรรดทัดใน MR (Inline Comments) เพื่อให้คุณพิจารณาตัดสินใจทันที

## 2. โครงสร้างและความหมายของข้อความแจ้งเตือน (Understanding AI Comments)

เมื่อ AI ตรวจพบสิ่งผิดปกติหรือประเด็นความมั่นคงปลอดภัย จะปรากฏคอมเมนต์ในลักษณะดังตัวอย่างนี้:

> 🤖 **[AI Security Assistant: Confirmed Vulnerability]**
>
> * **ระดับความรุนแรง (Severity):** 🔴 High
> * **ปัญหาหลัก (Vulnerability):** Broken Access Control / IDOR (CWE-639)
> * **คำอธิบายบริบท:** พบการรับตัวแปร `userId` จาก URL parameter โดยตรงในบรรทัดที่ 42 เพื่อนำไปดึงประวัติทางการเงิน โดยไม่มีการตรวจสอบสิทธิ์ว่าผู้ใช้คนนั้นเป็นเจ้าของบัญชีจริงหรือไม่ (แม้ว่า SAST ด่านแรกจะตรวจไม่พบประเด็นนี้เนื่องจากไม่ได้เป็น Sql Injection แต่เป็นช่องโหว่เชิงตรรกะ)
> * **แนวทางการแก้ไข (Suggested Fix):**
>   ```typescript
>   // ❌ โค้ดเดิมที่มีปัญหา
>   const userFinance = await FinanceService.getByUserId(req.params.userId);
>
>   // ✅ แนะนำให้เปลี่ยนเป็นตรวจสอบ Token สิทธิ์ก่อน
>   if (req.user.id !== req.params.userId && req.user.role !== 'admin') {
>       return res.status(403).json({ error: 'Permission Denied' });
>   }
>   const userFinance = await FinanceService.getByUserId(req.params.userId);
>   ```
> * **การอ้างอิงมาตรฐาน:** [CWE-639: Access Control Bypass through User-Controlled Key](https://cwe.mitre.org/data/definitions/639.html)

## 3. การโต้ตอบและการคัดกรองการแจ้งเตือนผิดพลาด (Interacting with the Bot)

AI Bot ไม่มีความเชื่อถือระดับเด็ดขาด (Advisory Role) การจัดการกับการตอบกลับจึงขึ้นอยู่กับดุลยพินิจของทีมพัฒนา โดยมีแนวทางปฏิบัติ 3 รูปแบบ:

### 3.1 กรณีคำแนะนำถูกต้อง (Confirmed True Positive)
หากวิเคราะห์แล้วพบว่าปัญหาในข้อความแจ้งเตือนเป็นจริง:
- ทำการแก้ไขซอร์สโค้ดตามคำแนะนำ
- ทำการ `git commit` และ `git push` เข้ามาใหม่เพื่อรัน Pipeline ตรวจเช็กอีกครั้ง

### 3.2 กรณีเป็นการแจ้งเตือนที่ผิดพลาด (False Positive)
หากพิจารณาแล้วพบว่าส่วนโค้ดดังกล่าวมีความปลอดภัยอยู่แล้วในเลเยอร์อื่น หรือไม่ได้เป็นช่องโหว่จริง:
1. กดตอบคอมเมนต์ของ AI ใน GitLab MR
2. พิมพ์คำสั่งยืนยัน เช่น `/dismiss-fp [พร้อมเขียนอธิบายสั้นๆ ว่าเหตุใดจึงไม่เป็นประเด็นความเสี่ยง]`
3. ระบบจะบันทึกข้อความของคุณลงในฐานข้อมูลความจำเพื่อสอนให้ AI ฉลาดขึ้นในรอบการวิเคราะห์ถัดไป และเปิดทางให้ทำการรีวิวต่อได้

### 3.3 การสั่งให้บอทเริ่มรีวิวใหม่อีกครั้ง (Force Re-review)
หากต้องการให้บอทคำนวณและประเมินผลการตรวจสอบ MR ใหม่อีกรอบหลังมีการแก้ไข:
- สามารถพิมพ์คำสั่ง `/ai-re-review` ในคอมเมนต์หลักของ Merge Request
- GitLab Runner จะสั่งสตาร์ทด่าน AI รีวิวใหม่อีกครั้งทันทีโดยไม่ต้องรันด่าน SAST ใหม่ทั้งหมด

## 4. การเขียนชุดรหัสสั่งละเว้นสำหรับ AI (Code Annotations)

กรณีต้องการบล็อกการแจ้งเตือนของ AI บนโค้ดบางจุดตั้งแต่ระดับซอร์สโค้ด สามารถใช้ Syntax เฉพาะได้ดังนี้:

- **สำหรับภาษา JavaScript/TypeScript:**
  ```javascript
  // ai-ignore-next-line: อธิบายสาเหตุ เช่น มีการใช้ระบบความปลอดภัยเช็กผ่าน Middleware อื่นแล้ว
  const unsafeData = req.body.input;
  ```
- **สำหรับภาษา Python:**
  ```python
  unsafe_query = db.execute(sql)  # ai-ignore-line: มีการ Sanitized ค่าในฟังก์ชัน helper แล้ว
  ```
- **สำหรับภาษา HTML/CSS:**
  ```html
  <!-- ai-ignore: จุดนี้เป็น Mockup URL ปลอดภัยจากการโจมตี XSS -->
  <div id="user-display"></div>
  ```

> [!WARNING]
> **ข้อควรระวัง:** การใช้คำสั่งละเว้น `ai-ignore` จะต้องมีการบันทึกเหตุผลประกอบที่ชัดเจน และหากเป็นการสแกนระดับการส่งขึ้น Production ทีม Tech Lead จะต้องมีการตรวจสอบคำร้องนี้ด้วยตนเองเสมอ
